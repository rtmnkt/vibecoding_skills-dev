# .docs/requirements/ — 要件駆動開発ガイド

## 概要

このフォルダの `.md` ファイルが開発の核。ユーザーが要件を列挙し、エージェントが実装・検証する。

## ファイル規約

- ファイル名: `XXXX.md` (4桁ゼロ埋め連番, 0001〜)
- 0000.md: 本ファイル（メタ情報）

## フォーマット

### front matter (YAML)

```yaml
---
passed_stutas:          # ※ユーザー命名。typoではない。変更不可。
  YYYY-MM-DD:           # 検証実施日
    - reqno: status     # "passed" or 検証結果の補足文字列
  YYYY-MM-DD:           # 別の日に再検証した場合、日付を追加
    - reqno: status
---
```

status値:
- `passed` — 問題なく通過
- 補足文字列 — 通過したが条件・制限あり（例: `"LLMバックエンドはテキストのみ"`）
- 日付エントリが無い要件 = 未検証

### 本文

```
10. 要件の説明文
20. 別の要件
21. 20の子要件（10刻みの親に対し1刻みで子を振る）
```

- 番号は10刻み。子要件は親の1刻み (例: 70の子→71,72,73,74。80の子→81,82)
- ネストは1段まで（子の子は作らない。必要なら別ファイルに分離）
- エージェントが追記する場合、末尾の最大番号の次の10刻みから振る（末尾が80なら90〜）
- 日本語で書く。1行1要件。簡潔に。

## エージェントのワークフロー

1. **ユーザーが要件ファイルを作成 or 更新する**
2. **エージェントは要件を読み、検証する**
   - コード読解で判断できるもの → grepやファイル確認で検証
   - 実行しないと分からないもの → ビルド+実行で検証
   - 既に満たされている → `passed_stutas` に日付+結果を追記
   - 未対応 → 計画 → 実装 → 再検証 → `passed_stutas` 追記
3. **エージェントが暗黙の要件を発見した場合、本文末尾に追記する**
   - ユーザーが不要と判断すれば勝手に消す。追記を躊躇しないこと。
4. **対応する `.docs/specs/XXXX.md` に実装コンテキストを記録する**（実装完了時に作成・更新）

## リグレッション

- 過去の `passed_stutas` エントリは消さない（履歴として保持）
- 再検証で失敗した場合、新しい日付で失敗メモを追記する
  ```yaml
  2026-04-10:
    - 30: "REGRESSION: xcap v0.8でBitBlt→WGCに変更されキャプチャ除外が効かなくなった"
  ```

## requirements/ と specs/ の関係

| フォルダ | 読者 | 内容 | 書き手 |
|----------|------|------|--------|
| requirements/ | 人間+エージェント | 何を満たすべきか（WHAT） | ユーザー主体 |
| specs/ | エージェント | どう実装したか・ハマりポイント（HOW/WHY） | エージェント主体 |

specs/ の詳細は `.docs/specs/0000.md` 参照。

## ユーザーからの指示事項

- `passed_stutas` のスペルは固定。直すな。
- 要件ファイルの内容はユーザーが自由に書き換える。エージェントが書いたものを消されても気にしないこと。
- 問題があればユーザーに提案・警告せよ。遠慮不要。
- サブエージェントに haiku モデルを使うな。品質が低い。
- rubber-duck サブエージェントを非自明なタスクで活用せよ。
